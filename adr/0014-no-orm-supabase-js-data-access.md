# 0014. No ORM — `supabase-js` is the application data-access layer

- **Status:** Accepted
- **Date:** 2026-06-09
- **Source:** datasource-be-foundation epic (C1)

## Context

With Supabase as the database engine and SQL-first migrations (ADR-0012,
ADR-0013), we need a data-access strategy for application code. The prior plan
to adopt Drizzle ORM was deferred and never implemented.

## Decision

**No ORM.** The application data-access layer is `supabase-js`, with TypeScript
types generated by `supabase gen types typescript`. A thin query/SQL layer
(likely Postgres functions exposed via RPC) is re-evaluated **per-domain, only
on real query friction** (e.g. transactional Finance logic) — not adopted
speculatively.

This decision **supersedes** the previously-deferred Drizzle ORM plan.

## Consequences

- `supabase-js` handles CRUD, auth, storage, and real-time with a single client.
- Generated types keep application code in sync with the schema without manual
  type definitions.
- Complex queries that don't map well to the PostgREST API surface are handled
  via Postgres functions (`rpc()`), not an ORM query builder.
- No Drizzle, Prisma, or Knex dependency — fewer abstractions, less indirection.
- Trade-off: developers write more SQL for complex cases. This is intentional —
  SQL is the lingua franca of the Supabase ecosystem.

## Alternatives considered

- **Drizzle ORM** — excellent TypeScript integration, but adds a parallel
  migration system and query builder that competes with `supabase-js`. Rejected
  to keep one data-access path.
- **Prisma** — heavy runtime, introspection model conflicts with Supabase CLI
  ownership of migrations.
- **Knex** — query builder without ORM, but `supabase-js` already covers most
  query patterns and adds auth/real-time/storage.
