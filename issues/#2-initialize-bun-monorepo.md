## 1. TL;DR (read this first)

Create the **empty-but-correct skeleton** of the NafiOS monorepo: the top-level folder layout from the guide, the root `package.json` with Bun workspaces, a strict `tsconfig.base.json`, a `.gitignore`, and a generated `bun.lockb`.

This ticket builds the **shape of the repo**, not its contents. When it's done, `bun install` runs green on a fresh clone and the directory tree matches the guide — but the files inside (`CLAUDE.md` content, specs, ADRs, lint config, the example package) are still stubs owned by later tickets.

If a later ticket ever asks "where do I put this folder / how is the workspace wired?", the answer exists because of A1.

---

## 2. Context / why

Per [bun-monorepo-setup.md](../temp_docs/monorepo-setup-guide.md), an AI-native repo assumes a stateless agent that wakes up with zero context. The first thing that agent needs is a **predictable, discoverable top-level layout** and a workspace it can install and resolve. A1 turns the guide's "Top-Level Repository Layout" and "Workspace Configuration" sections into real files.

We deliberately do the skeleton _first and alone_ so that every subsequent ticket (CLAUDE.md authoring, specs, generators, UI kit, datasource) has a known, stable place to land — no ticket has to re-decide folder structure.

---

## 3. Scope

### In scope ✅

- Top-level directory tree from the guide, with empty dirs tracked (`.gitkeep`).
- Root `package.json`: `private`, `workspaces`, Bun version pin, minimal script _placeholders_.
- `tsconfig.base.json` with strict settings, intended to be extended by each package.
- Root `.gitignore`.
- Generated `bun.lockb` (a product of `bun install`, not hand-written).
- **Placeholder stubs** for the root files named in the layout (`CLAUDE.md`, `AGENTS.md`, `README.md`) so the tree is complete — content is explicitly deferred.

### Out of scope ❌ (owned elsewhere)

- Real `CLAUDE.md` / `.claude/context/*` content → **D1**
- Working `bun run check` (typecheck + lint + format + test) → **A3**
- Biome / lint+format config → **C1**
- The canonical example package (`packages/core-utils`) → **A2**
- Specs system, ADR system, templates → **B1 / B2**
- Generators, slash commands, skills → **D2 / D3**
- CI workflows, CODEOWNERS, PR template → **C3**

> Rule of thumb: if it's a _folder or workspace wiring_, it's A1. If it's _content that goes inside a folder_, it belongs to that folder's ticket.

## 4. Tasks

### 4.1 Create the top-level tree

Match the guide's layout exactly. Add `.gitkeep` to dirs that will be populated by later tickets so git tracks them:

```
nafios/
├── .claude/              # .gitkeep (D1 fills this)
├── .github/              # .gitkeep (C3 fills this)
├── apps/                 # .gitkeep
├── packages/            # .gitkeep
├── services/             # .gitkeep
├── tooling/              # .gitkeep
├── docs/                 # .gitkeep
├── specs/                # .gitkeep (B1 fills this)
├── adr/                  # .gitkeep (B2 fills this)
├── .agent/
│   ├── scratch/          # gitignored
│   ├── plans/            # .gitkeep
│   └── reports/          # .gitkeep
├── CLAUDE.md             # STUB → real content in D1
├── AGENTS.md             # STUB → mirrors/points to CLAUDE.md
├── README.md             # STUB → human entry point, expanded later
├── .gitignore
├── package.json
├── tsconfig.base.json
└── bun.lockb             # generated by `bun install`
```

> `biome.json` and `turbo.json` from the guide are **intentionally omitted here** — see §6.

### 4.2 Root `package.json`

Workspace root only; scripts are placeholders that A3/C1 will implement.

```jsonc
{
  "name": "nafios",
  "private": true,
  "workspaces": ["apps/*", "packages/*", "services/*", "tooling/*"],
  "engines": { "bun": ">=1.1.0" }, // confirm against the Bun version we standardize on
  "scripts": {
    // Real implementations land in A3 (check) / C1 (lint+format).
    // Kept as no-ops so the keys exist and the workspace installs cleanly.
    "check": "echo 'check is implemented in ticket A3' && exit 0",
    "typecheck": "echo 'typecheck wired in A3' && exit 0",
    "lint": "echo 'lint wired in C1' && exit 0",
    "test": "echo 'test wired in A3' && exit 0",
  },
}
```

### 4.3 `tsconfig.base.json`

A strict baseline each package extends. Treat this as a starting point.

```jsonc
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true,
    "verbatimModuleSyntax": true,
    "isolatedModules": true,
    "moduleDetection": "force",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "target": "ESNext",
    "lib": ["ESNext"],
    "types": ["bun-types"], // confirm current Bun type package (bun-types vs @types/bun)
    "skipLibCheck": true,
    "resolveJsonModule": true,
    "allowImportingTsExtensions": true,
    "noEmit": true,
  },
}
```

### 4.4 `.gitignore`

```gitignore
# deps
node_modules/

# build output
dist/
build/
*.tsbuildinfo

# claude code (personal overrides)
.claude/settings.local.json

# agent scratch (everything else under .agent/ is tracked)
.agent/scratch/

# env
.env
.env.*
!.env.example

# os / editor / misc
.DS_Store
*.log
coverage/
```

> Note: **`bun.lockb` is committed** — do not add it to `.gitignore`.

### 4.5 Root stub files

- `CLAUDE.md`, `AGENTS.md`, `README.md` each contain one line stating they're placeholders and naming the ticket that fills them (e.g. `<!-- STUB: real content authored in ticket D1 -->`). This keeps the tree complete without pre-empting the owning ticket.

### 4.6 Install & verify

- Run `bun install`; confirm it succeeds and produces `bun.lockb`.

- [ ] Top-level directory tree matches the guide's "Top-Level Repository Layout" (verified by `ls`/tree).
- [ ] Empty future-dirs are tracked via `.gitkeep`; `.agent/scratch/` is gitignored.
- [ ] Root `package.json` declares `private: true`, the four workspace globs, and a Bun version pin.
- [ ] `tsconfig.base.json` exists with strict settings and is documented as the base others extend.
- [ ] `.gitignore` present; `bun.lockb` is **not** ignored.
- [ ] `CLAUDE.md`, `AGENTS.md`, `README.md` exist as clearly-marked stubs pointing to their owning tickets.
- [ ] **A fresh clone runs `bun install` green with zero manual steps**, producing `bun.lockb`.
- [ ] No content that belongs to A2/A3/B/C/D/E/F has leaked into this ticket.

## 6. Decisions to flag (candidate ADRs)

A1 surfaces two structural choices the guide leaves open. Recommend recording each as an ADR in **B2** rather than deciding silently here:

1. **Task runner: Bun-native `--filter` vs Turborepo.** The guide's root `package.json` already uses `bun --filter '*'`, which suggests we may not need `turbo.json` at all. Recommendation: **start Bun-native, defer/skip `turbo.json`** unless caching/orchestration pain appears. → ADR candidate.
2. **Workspace type resolution: tsconfig `paths` aliases vs Bun workspace resolution.** Bun resolves `@nafios/*` from each package's `name` field, so explicit `paths` in `tsconfig.base.json` may be redundant (or even drift-prone). Recommendation: **rely on workspace resolution; add `paths` only if editor/typecheck needs it.** → ADR candidate.

## 8. Notes for the implementing agent

- Read the guide sections **"Top-Level Repository Layout"** and **"Workspace Configuration"** before starting.
- **Do not** hand-create `bun.lockb` — it is produced by `bun install`.
- **Do not** populate `.claude/`, `specs/`, `adr/`, or write real `CLAUDE.md` content — those are other tickets. Stubs + `.gitkeep` only.
- Confirm version-sensitive details against current Bun docs at implementation time: the `engines.bun` floor and the Bun **types package name** (`bun-types` vs `@types/bun`).
- When done, run `bun install` and paste the output; do not declare done until a fresh install is green.
